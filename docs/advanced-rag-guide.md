## 📥 写入阶段 (Indexing)

### 流程概览

```
原始文档
    ↓
┌─────────────┐
│  1. 预处理   │  格式转换、清洗、结构提取
└─────────────┘
    ↓
┌─────────────┐
│  2. 分块     │  按策略切分文档
└─────────────┘
    ↓
┌─────────────┐
│  3. 增强     │  提取关键词、生成摘要、识别实体
└─────────────┘
    ↓
┌─────────────┐
│  4. Embedding│  文本向量化
└─────────────┘
    ↓
┌─────────────┐
│  5. 存储     │  向量库 + Metadata
└─────────────┘
```

### 1. 预处理

**目的**：将原始文档转换为统一格式

| 处理项 | 说明 |
|--------|------|
| 格式转换 | PDF/Word/HTML → 纯文本/Markdown |
| 内容清洗 | 去除无效字符、修复编码 |
| 结构提取 | 识别标题、段落、列表 |

### 2. 分块 (Chunking)

**目的**：将长文档切分为适合检索的小块

#### 分块策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| `FIXED_SIZE` | 按固定字符数切分 | 通用场景 |
| `RECURSIVE` | 从粗到细递归切分 | 复杂文档 |
| `MARKDOWN` | 按Markdown结构切分 | 结构化文档 |
| `SEMANTIC` | 按语义相似度切分 | 高质量要求 |
| `AGENTIC` | LLM智能切分 | 高价值文档 |

#### 分块参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `chunkSize` | 目标块大小 | 300-600字符 |
| `minChunkSize` | 最小块大小 | 100字符 |
| `maxChunkSize` | 最大块大小 | 1500字符 |
| `overlap` | 重叠大小 | 50-100字符 |

#### API 调用

```bash
POST /api/v1/chunking/chunk-and-store
{
  "content": "文档内容...",
  "documentName": "员工入职流程",
  "collection": "hr_knowledge",
  "preset": "SOP",
  "strategy": "MARKDOWN"
}
```

### 3. 增强处理

**目的**：丰富 chunk 的元信息，提升检索效果

| 增强项 | 说明 | 实现 |
|--------|------|------|
| 关键词提取 | 提取 chunk 的核心关键词 | TF-IDF / 词频统计 |
| 类型识别 | 识别 chunk 类型（概述/步骤/FAQ） | 规则匹配 / LLM |
| 实体提取 | 提取人名、地名、术语等 | NER / 正则 |
| 摘要生成 | 生成 chunk 摘要 | LLM |

### 4. Embedding

**目的**：将文本转换为向量表示

#### 向量类型

| 类型 | 说明 | 适用场景 | 实现方式 |
|------|------|---------|----------|
| **稠密向量** | 语义向量，捕获文本语义关系 | 语义搜索、相似性匹配 | Embedding模型生成 |
| **稀疏向量** | 词频向量，基于词汇匹配 | 关键词搜索、精确匹配 | BM25/TF-IDF算法 |

#### 支持的模型

| 模型 | 维度 | 特点 |
|------|------|------|
| OpenAI text-embedding-3-small | 1536 | 通用、高质量 |
| OpenAI text-embedding-3-large | 3072 | 更高精度 |
| 阿里 DashScope | 1536 | 中文优化 |
| 本地模型 | 可变 | 私有部署 |

#### 混合检索策略

根据文档类型自动选择检索策略：

| 文档类型 | 检索策略 | 配置 | 说明 |
|----------|----------|------|------|
| **标准文档** (SOP/知识库) | 稠密向量 | `enableSparseVector=false` | 仅语义检索，依赖tags过滤 |
| **非标准文档** (FAQ/产品文档) | 混合检索 | `enableSparseVector=true` | 稠密+稀疏混合检索 |

#### 注意事项

- **模型一致性**：写入和查询必须使用相同的 Embedding 模型
- **Token限制**：注意模型的最大 token 限制
- **批量处理**：大量文档建议批量 Embedding
- **稀疏向量配置**：通过 `ChunkingConfig.enableSparseVector` 控制

### 5. 向量存储

**目的**：持久化存储向量和元数据

#### 存储结构

```json
{
  "id": "doc-001-chunk-0",
  "vector": [0.1, 0.2, ...],
  "sparse_vector": {"入职": 0.8, "材料": 0.6, "准备": 0.7},
  "content": "chunk 原文内容",
  "metadata": {
    "document_name": "员工入职流程",
    "chunk_type": "step",
    "step_number": 1,
    "keywords": ["入职", "材料", "准备"],
    "tenant_id": "tenant-001",
    "collection": "hr_knowledge",
    "embedding_model": "text-embedding-3-small",
    "enable_sparse_vector": true,
    "sparse_analyzer": "chinese"
  }
}
```

#### 索引策略

| 索引类型 | 向量类型 | 说明 | 适用场景 |
|---------|----------|------|---------|
| FLAT | 稠密 | 精确搜索 | 小数据量 |
| IVF | 稠密 | 倒排索引 | 中等数据量 |
| HNSW | 稠密 | 图索引 | 大数据量、高性能 |
| SPARSE_INVERTED | 稀疏 | 倒排索引 | 关键词搜索 |

## 🔍 查询阶段 (Retrieval)

### 流程概览

```
用户查询
    ↓
┌─────────────┐
│ 1. 查询改写  │  优化查询表达
└─────────────┘
    ↓
┌─────────────┐
│ 2. 向量检索  │  相似度搜索
└─────────────┘
    ↓
┌─────────────┐
│ 3. 重排序    │  精细化排序
└─────────────┘
    ↓
┌─────────────┐
│ 4. 上下文补全│  补充关联信息
└─────────────┘
    ↓
┌─────────────┐
│ 5. LLM生成   │  生成最终答案
└─────────────┘
```

### 1. 查询改写 (Query Rewriting)

**目的**：将用户的口语化问题转换为更适合检索的查询

#### 改写策略

| 策略 | 说明 | 示例 |
|------|------|------|
| 关键词提取 | 提取核心概念 | "怎么入职" → "入职流程 步骤" |
| 同义词扩展 | 添加同义词 | "入职" → "入职 报到 报道" |
| 问题分解 | 拆分复杂问题 | "入职需要什么材料和流程" → ["入职材料", "入职流程"] |
| 上下文补充 | 结合历史对话 | 补充省略的主语/宾语 |

#### 实现方式

```
原始查询: "新员工第一天要干嘛"
    ↓
LLM 改写
    ↓
优化查询: "新员工入职第一天流程 步骤 材料准备"
```

### 2. 向量检索 (Vector Search)

**目的**：从向量库中检索相似文档

#### 检索参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `topK` | 返回结果数量 | 5-10 |
| `similarityThreshold` | 相似度阈值 | 0.7-0.8 |
| `filter` | 元数据过滤 | 按 collection、类型等 |

#### 检索优化

| 优化项 | 说明 |
|--------|------|
| 扩大检索范围 | 如果启用重排序，检索 topK*3 条 |
| 降低阈值 | 让重排序来筛选，避免漏召回 |
| 混合检索 | 向量检索 + 关键词检索 |

### 3. 重排序 (Reranking)

**目的**：对初步检索结果进行精细化排序

#### 重排序方法

| 方法 | 原理 | 特点 |
|------|------|------|
| Cross-Encoder | 查询-文档联合编码 | 精度高、速度慢 |
| LLM Rerank | LLM 评估相关性 | 灵活、成本高 |
| 规则重排 | 基于元数据规则 | 简单、可控 |

#### 重排序流程

```
初步检索结果 (30条)
    ↓
Cross-Encoder 打分
    ↓
按相关性排序
    ↓
取 Top K (10条)
```

### 4. 上下文补全 (Context Completion)

**目的**：补充检索结果中缺失的关联信息

#### 补全策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 前后文补全 | 获取 chunk 的前后相邻块 | 步骤类文档 |
| 父节点补全 | 获取上级章节内容 | 层次结构文档 |
| 关联文档补全 | 获取引用的其他文档 | 跨文档引用 |
| 条件分支补全 | 获取相关的条件分支 | SOP 文档 |

#### 补全流程

```
检索到: 步骤3
    ↓
检查是否需要补全
    ↓
补充: 步骤2、步骤4（前后文）
    ↓
补充: 前置条件（父节点）
```

### 5. LLM 生成 (Generation)

**目的**：基于检索结果生成最终答案

#### Prompt 构建

```
System Prompt:
你是一个专业的知识库助手。请根据提供的上下文回答用户问题。
要求：
1. 严格基于上下文回答，不要编造
2. 如果是步骤类问题，按步骤清晰列出
3. 引用来源文档
4. 如果信息不足，明确说明

Context:
[检索到的文档内容]

User Query:
[用户问题]
```

#### 生成参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `temperature` | 随机性 | 0.1-0.3 |
| `max_tokens` | 最大输出长度 | 1000-2000 |
| `model` | 模型选择 | gpt-4 / gpt-3.5-turbo |

## 🔧 API 使用

### 写入 API

#### 分块并存储

```bash
POST /api/v1/chunking/chunk-and-store
Content-Type: application/json
Authorization: Bearer <token>

{
  "content": "# 员工入职流程\n\n## 概述\n...",
  "documentName": "员工入职流程",
  "collection": "hr_knowledge",
  "preset": "SOP"
}
```

#### 直接存储（已分块）

```bash
POST /api/v1/vectors/store
Content-Type: application/json
Authorization: Bearer <token>

{
  "collection": "hr_knowledge",
  "documents": [
    {
      "id": "doc-001-chunk-0",
      "content": "chunk 内容",
      "metadata": {
        "document_name": "员工入职流程",
        "chunk_type": "overview"
      }
    }
  ]
}
```

### 查询 API

#### Advanced RAG 查询

```bash
POST /api/v1/rag/query
Content-Type: application/json
Authorization: Bearer <token>

{
  "collection": "hr_knowledge",
  "query": "新员工入职第一天需要做什么",
  "mode": "ADVANCED",
  "topK": 5,
  "enableQueryRewrite": true,
  "enableRerank": true,
  "enableContextCompletion": true,
  "model": "gpt-4"
}
```

#### 响应示例

```json
{
  "success": true,
  "answer": "新员工入职第一天需要完成以下步骤：\n\n1. **材料准备**：携带身份证、学历证书...\n2. **HR登记**：前往HR部门...\n\n【来源：员工入职流程-步骤1、步骤2】",
  "sources": [
    {
      "id": "doc-001-chunk-1",
      "content": "步骤1: 材料准备...",
      "score": 0.92
    }
  ],
  "rewrittenQuery": "新员工入职第一天流程步骤",
  "mode": "ADVANCED"
}
```

## 🎯 最佳实践

### 写入阶段

1. **选择合适的分块策略**：SOP用MARKDOWN，通用文档用RECURSIVE
2. **控制chunk大小**：300-600字符为宜
3. **丰富metadata**：便于后续过滤和排序
4. **保持模型一致**：记录使用的Embedding模型

### 查询阶段

1. **启用查询改写**：提升召回率
2. **启用重排序**：提升精确度
3. **合理设置topK**：平衡召回和性能
4. **使用上下文补全**：保持信息完整性

## 📊 RAG 模式对比

| 模式 | 流程 | 适用场景 |
|------|------|---------|
| **NAIVE** | 检索 → 生成 | 简单问答 |
| **ADVANCED** | 改写 → 检索 → 重排 → 补全 → 生成 | 复杂问答 |
| **GRAPH** | 检索 → 图遍历 → 生成 | 关系推理 |
